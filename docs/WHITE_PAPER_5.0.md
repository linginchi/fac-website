## 貳、架構 5.0 核心原則 (Core Principles)

### 2.1 身分與術語「去二元化」
- **甲方 (Party A)**：服務需求方、付錢的一方。不再使用「僱主 / Employer」等易與勞動法混淆的稱謂。
- **乙方 (Party B)**：服務提供方、具備技能並收取服務費的一方。不再使用「專家 / Expert」作為帳號永久標籤。
- **身分流動化**：用戶帳號統一化，其身分由**當下意圖**決定。同一用戶可在此任務為甲方、彼任務為乙方；嚴禁以 `isEmployer` / `isExpert` 等布爾值固化角色。

### 2.2 能力導向：原子化技能矩陣
- **去行業化匹配**：將用戶特長拆解為**原子技能標籤**（如：談判、Python 開發、法律審計、教學），不再以粗粒度「行業」作為匹配維度。
- **技能分類**：Hard（認證/技術）、Soft（軟技能）、Industry（行業知識），每項帶權重 1–5。
- **數據結構**：乙方維護 **CapabilityProfile**（能力檔案）；甲方維護 **RequirementProfile**（需求檔案）。兩者並存於同一用戶帳號，依情境切換使用。

### 2.3 意圖分流引擎與萬能框 (OmniBox)
- **萬能框**為平台唯一入口。根據用戶輸入自動判斷當前行為模式：
  - **「找 / 需要…」**（甲方意圖）→ 導向需求解析與乙方匹配。
  - **「我能 / 我是…」**（乙方意圖）→ 更新 CapabilityProfile，進入能力展示與匹配。
- **Logo（港匠匯）點擊**：清除當前臨時意圖狀態，重置回首頁萬能框，確保導航可預期。

### 2.4 隱私與安全 (Privacy First)
- **資訊揭露控制**：默認隱藏所有用戶真實聯繫方式。唯有在**合約初步意圖達成**、且用戶手動點擊「**授權披露**」後，方可展示敏感資訊。
- **物理路由攔截**：所有私有路徑（/vault、/wallet、/profile 等）在**頁面渲染前**即完成強制重定向至 /login；未登錄者不得訪問。

### 2.5 契約與信任：智能擔保交易
- **30% 訂金鎖定 (Escrow)**：合約確認後，甲方支付的 30% 訂金由平台鎖定，確保乙方開工保障；尾款 70% 於甲方確認交付後釋放。
- **自雇人士服務協議**：在所有環節自動附帶《自雇人士服務協議》，明確**無 MPF、無雇傭義務**，乙方為獨立服務提供者。

### 2.6 公益與透明運作
- **非營利費用模型**：移除佣金抽成。僅收取 **Platform_Fee**（覆蓋算力與維護成本），並為**年度審計看板**預留數據接口。
- **營運餘額**：法幣收費在扣除必要成本後，餘額撥入「香港專業人才傳承基金」。

## 參、去中心化技術與數據主權 (Web3 Infrastructure)

### 3.1 去中心化保險箱
- 用戶的資歷與案例存儲於**專屬加密節點**。
- 非經本人授權，連實驗室管理人員也無法查看。
- 技術架構：分散式加密存儲 + 零知識證明 (ZKP)。

### 3.2 智慧資產冷備份
- 支持 Executive 級別用戶將智慧資產導出至物理冷錢包。
- 導出格式：RSA 加密 JSON，支持離線解密驗證。

### 3.3 非盈利運營承諾
| 承諾項目 | 內容 |
|---------|------|
| 董事義務職 | 全體董事以無薪義務形式履行職責 |
| 利潤不分配 | 法幣收費餘額全數撥入「香港專業人才傳承基金」 |
| 技術去中心化 | 用戶數據存儲於用戶專屬加密節點 |
| 公正撮合 | 匹配算法以**原子技能相關性**為核心標準 |
| 無佣金 | Platform_Fee 僅覆蓋算力與維護，不抽成 |

## 肆、雙向對接模式 (Bidirectional Engagement) — 5.0 表述

### 4.1 乙方側（服務提供方 / Party B）
- 透過萬能框輸入「我能 / 我是…」或上傳簡歷，建立/更新 **CapabilityProfile**。
- 原子技能自動提取並寫入能力檔案；可設置服務費率、可用時段。
- 語音優先：動口不手動，一句話完成能力登記。

### 4.2 甲方側（服務需求方 / Party A）
- 透過萬能框輸入「找 / 需要…」，觸發需求解析與**RequirementProfile**（或 JD 解析）。
- AI 依原子技能匹配乙方；解碼脫敏資歷可消耗 $FAC。
- 意圖達成後進入擔保交易流程（訂金鎖定 → 授權披露 → 開工 → 交付 → 尾款釋放）。

## 柒、契約與擔保交易 (Escrow & Self-Employed Agreement)

### 7.1 合約狀態流轉
```
draft → intent_matched → deposit_locked → in_progress → delivered → completed
```
- **intent_matched**：甲乙方均確認對接，可進行「授權披露」。
- **deposit_locked**：30% 訂金已鎖定，乙方可開工。
- **completed**：甲方確認交付，70% 尾款釋放。

### 7.2 《自雇人士服務協議》強制附帶
- 乙方為獨立服務提供者，非甲方受僱人士。
- 無 MPF、無《僱傭條例》義務；乙方自行承擔稅務申報。
- 雙方點擊「確認合約」即視為接受上述條款。

## 捌、隱私與授權揭露 (Privacy & Disclosure)

### 8.1 默認隱藏
- 所有用戶聯繫方式、敏感個人資料默認**不展示**。
- API 與前端渲染層均須檢查**授權揭露**狀態。

### 8.2 授權揭露觸發條件
- 僅當合約狀態 ≥ **intent_matched** 時，甲/乙方**各自手動**點擊「授權披露」後，對方方可見聯繫方式。
- 揭露狀態以合約為粒度記錄（partyAContactDisclosed / partyBContactDisclosed）。
